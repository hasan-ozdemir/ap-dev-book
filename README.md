# Prodyum – Development Playbook

## Introduction

Welcome to **devyum**, the definitive playbook for every Prodyum team building secure, resilient, and elegant digital products. The portal curates the best of our engineering craft—covering environment setup, modern C# fluency, architecture decisions, .NET MAUI delivery, cloud operations, and program governance—so squads can onboard fast, deliver consistently, and scale with confidence.

For engineering leaders and senior developers, the playbook accelerates delivery with runnable samples, reference implementations, control catalogs, performance guides, and DevOps blueprints that align with .NET 9 and .NET MAUI best practices. Product managers and business stakeholders gain traceability through clear architecture trade-offs, release checklists, compliance-ready store submissions, and operations runbooks. Executive sponsors see the breadth of organizational readiness: talent enablement paths, continuous delivery pipelines, platform governance, cloud cost controls, and innovation accelerators such as the Developer Knowledge Base and AI blueprint templates.

The content is curated first for Prodyum engineers so squads share a common language, engineering standards, and culture of craftsmanship. At the same time, the guidance is intentionally open and reusable: any developer, partner, or community team tackling cross-platform delivery can leverage these modules as a complementary companion to their own playbooks.

Use this site as the launchpad for every initiative—start with the numbered Table of Contents to build foundational knowledge, then dive into specialized modules for architecture, platform development, DevOps automation, cloud services, testing, delivery operations, and cross-functional collaboration. Every article is sourced, actionable, and cross-linked, making it easy to navigate from zero to production hero.

## Why it matters

- **Business value**
  - Shortens time-to-market for partner initiatives through repeatable blueprints.
  - Increases transparency across engineering, product, and business stakeholders.
  - Reduces launch risk by codifying store-readiness, compliance, and support playbooks.
- **Developer value**
  - Provides runnable samples, tooling guides, and decision matrices for daily work.
  - Documents conventions for code quality, testing, and collaboration.
- Offers migration paths (Xamarin.Forms -> .NET MAUI, legacy .NET -> modern .NET).

## How to use the playbook

1. Scan the **Table of Contents** to locate the area that matches your current task.
2. Refresh fundamentals in **Foundations**, then dive into topic-specific sections (MAUI, DevOps, Azure, Delivery Operations, AI blueprints).
3. Refer back to this README for a single entry point - the table of contents links to every article in the repository.
4. Contribute improvements through pull requests; guidelines are listed below.

## Repository structure & tooling

The portal follows a docs-as-code layout so engineering, product, and operations content evolve alongside the code while staying reviewable through pull requests.citeturn1search2 Core directories and automation entry points include:

```
docs/
  landing/                   -> Vision, glossary, and roadmap
  00-foundations/            -> C#, .NET, tooling fundamentals
  01-architecture/           -> Solution planning and patterns
  02-mobile-maui/            -> MAUI delivery playbooks
  04-devops/                 -> CI/CD and automation practices
  05-cloud-azure/            -> Cloud computing and Azure services
  06-delivery-operations/    -> Store publishing & support
  07-practitioner-toolbox/   -> Checklists, templates, practitioner guides
  08-mobile-ai-blueprints/   -> AI proof-of-concept catalog
  09-knowledge-base/         -> Curated Q&A reference
samples/
  ConsoleSnippets/           -> Runnable C# sample snippets
  TodoApp/                   -> Showcase .NET MAUI sample
scripts/                     -> Build/test/publish helpers (link validation, automation)
.github/workflows/           -> Continuous integration & publishing jobs
mkdocs.yml                   -> Static site configuration and navigation map
AGENTS.md                    -> Contributor operating procedures
```

MkDocs Material powers the published site, and the validation scripts keep navigation, links, and build outputs healthy before every merge.citeturn0search1turn1search2

### Authoring guidelines

- Every Markdown file begins with YAML front matter (title, description, last_reviewed, owners) so the portal stays maintainable and searchable.citeturn1search2
- Prefer concise, active-voice sentences and structured lists to improve scannability for technical readers.citeturn1search6
- Pair conceptual explanations with runnable samples (`dotnet build`, `dotnet run`, `dotnet test`) wherever feasible to reinforce learning by doing.citeturn1search2
- Attribute every external fact with an inline citation so readers can verify the source material and stay compliant with documentation governance.citeturn1search2turn1search6

### Local development

1. Install Python 3.11+ and `pipx` (or ensure they’re on PATH).citeturn0search1
2. Provision the documentation toolchain with `pipx install mkdocs-material`, which installs MkDocs and the Material theme in an isolated environment.citeturn0search1
3. Run `mkdocs serve` for a live preview with hot reload while editing content.citeturn0search1
4. Execute `pwsh scripts\validate-docs.ps1` to build the site with MkDocs and run the link checker; append `-SkipLinks` when network access is limited.citeturn0search1turn1search2
5. Use `python scripts/check_links.py` for quick external link audits between larger validation runs so regressions surface early.citeturn1search2

### Contribution workflow
